The Grid Not intentionally a Tron reference

Essence

The 960 Grid System is an effort to streamline web development workflow by providing commonly used dimensions, based on a width of 960 pixels. There are two variants: 12 and 16 columns, which can be used separately or in tandem. Read more.

Dimension

The 12-column grid is divided into portions that are 60 pixels wide. The 16-column grid consists of 40 pixel increments. Each column has 10 pixels of margin on the left and right, which create 20 pixel wide gutters between columns. See this in action.

Source Order

By utilizing the push-XX and pull-XX classes, elements can be rearranged, independent of the order in which they appear in the markup. This allows you to keep more pertinent info higher in the HTML, without sacrificing precision in your page layout. For instance, view the source code of this site to see how the #content area has been re-positioned.

× The SEO team will appreciate that we place content before the sidebar in HTML even if if the site was designed differently.

Containers Two containers, one great taste

HTML

We use two grid variants, 12 and 16. To initiate one of the other is to start with a container. A container consists of HTML with a class that contains CSS properties that will create a block element, 960px wide.

<div class="container-12">
…
</div>

and:

<div class="container-16">
…
</div>

CSS

The CSS for this is simple, margin's for the left and right are set to auto which with center a 960px block level element. Position is set to relative in order to allow for absolutely positioned elements within a particular zone.

.container-12,
.container-16 { 
position:relative; 
margin:0px auto; 
width:960px; 
overflow:hidden;
text-align:left;
}

Code Commenting Be kind, rewind

HTML Commenting Standards

When there is a mess of HTML code and a ton of closing tags, sometimes it is not easy to figure out which closing <div> is the one we're looking for. Some say that commenting opening and closing HTML tags is necessary to help developers code read clearer. Although we agree with the sentiment, this is a bit much. We want the support team to easily traverse our code, and not just them, anybody who needs to edit code should be left well written, semantic, code.

We comment ending blocks where ever a class or ID tag was used. The syntax is simple, directly after the closing tag, start an HTML comment tag then add END and the classes or IDs in order in which they appear logically using periods for classes (.) and number sign for ID's (#) with spaces between each.

Example markup

Here is some code one might see for a Content & Sidebar zone…

<section id="content-area" class="container-12">
<article class="grid-8 content">
  …
</article><!-- END .grid-8 .content -->
<section class="grid-4 sidebar">
  …
</section><!-- END .grid-4 .sidebar -->
</section><!-- #content-area .container-12 -->

CSS Commenting Standards

Similarly we want to make our CSS comfortable to traverse for anybody besides ourselves who will be coming back to make edits. Remember this is more if a 'when' not 'if', people change their websites frequently and these need to be a flexible, and future proof as possible. The standard comment block has a line break before and after a line of comment. Also if there is something that may confuse another developer, write a comment, after the CSS attribute and value.

×Start and end your comment with three dashes /* --- --- */
×Think about what the presentational item does, how it works, and describe that as best you can in a comment.
/* --- Navigation --- */

.nav-inner {
background-color: #2c2c2c; /* Required for effect */
}

/* --- Content --- */

article .page-header {
padding-top:80px;
}

Push & Pull No physical contact required

Source Order

Sometimes the order in which we write our HTML is out of place with where something should presentationally display. We can rearrange the order of items in our document with a few classes. Some examples of this would be if a company logo is, presentationally, after less important information in the header. The 960.gs shows this well. If we view source on their landing page one will see the logo before the first content box about downloading, but the logo is pushed four columns to the right and the download box is pulled four columns to the left. Their code is slightly different than ours, we use dashes '-' not underscores '_' but it accomplishes the same effect.

Another example is the Card Law Firm. The layout for this site has the sidebar before the content. Well it's obvious we all want to get to the content before the sidebar, so we use the push and pull CSS tags to push the content (first in the HTML document) to the right, or past the columns required for the sidebar; and then pull, to the left, the sidebar (second in the HTML document) back past the content columns.

Pull

Pull code is the opposite of the push and starts similarly with pull- followed by the number of columns to the left one is pulling ones content. Back to the Card Law Firm, if we view source the sidebar, which comes after the content in the HTML is, being pulled twleve columns to the left to using CSS to move that sidebar before the content.

Example Code

<div class="grid-4 pull-12">
<section id="sidebar">
  …
</section><!-- END #sidebar -->
</div><!-- END .grid-4 .pull-12 -->

Push

The push code is straight forward, it starts with .push- followed by the number of columns to the right one is pushing ones content. Using Card Law Firm as an example, if we view the source of this site we see the content comes first in the HTML, and is being pushed four columns to the right.

Example Code

<div class="grid-12 push-4">
<header class="clearfix">
 …
</header><!-- END .clearfix -->
</div><!-- END .grid-12 .push-4 -->

Prefix & Suffix Offsetting columns

Padding

The prefix and suffix classes are used within the grid to add columns of padding before or after a grid area. So if a header is designed with a logo that takes up six grid columns, and the content for the right half of the sidebar had three columns that are unused, and the content starts again, this would be one example where one could use a prefix or a suffix to fill the gap. .prefix- followed by a number makes that numbers worth of columns in padding-left:; to move the content X columns via padding.

Likewise the opposite, .suffix- followed by the a number makes that numbers worth of columns in padding-right:; to add padding after content.

Alpha & Omega Similar to the first and last letters of the Greek alphabet, sort of…

Out of the gutters

Each column of the 12 or 16 grid system has 10px margin on the left and right called gutters. If one were to use .grid-2 this would take up two columns, absorbing the width of the gutters between those columns, and adding 10px margin-left and margin-right. If there are nested grid's within a design, then removing the extra margin will keep most designs from breaking to the next line. The last demo below is the result of not using .alpha or .omega for any of these nested grid elements, the last .grid-3 falls to the next line.
