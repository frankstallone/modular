Introduction Overview from 30,000 feet

The purpose of our Markup Guide is to give clear direction on the methodology to follow when performing front-end development to our products, and a reference guide, all in one place. We will be using examples of code where ever we can and request feedback for unclear items in order to make this markup guide better. Email me at frank.stallone@lexisnexis.com for feedback, clarity, or questions.

Starting with some general items, we'll explain some of the theory behind our Modular code, and we will drill into the nitty gritty CSS, Javascript, PHP. Throughout we'll discuss what we've done to make your developer job faster, easier, and painless for maintenance. Years of experience with HTML, CSS, and JavaScript have been melted into the best practices we use, so let's begin! =)

For examples we will use screenshots, code visuals, specific example pages, and finally Google Prettify to display code.

To the left is a general visual breakdown of our folder/file structure. Since we are doing child themes in WordPress we simply do not need anywhere near as many files. Whatever WordPress files we do not include in the child theme will automatically use the parent theme. The basis of this structure comes from the lovely gents at HTML5 Boilerplate. It has been adapted for WordPress themes, and to fit our needs with the design community, etc.

The !base-build folder is where the flat file templates are built for (typically) one inner page template and one home page template. This folder and contents never get uploaded to the server. Browser testing can be done easily, and JavaScript interaction can be tested as well. !base-build/html/ folder is where video impimentation with Shadowbox can be implimented. !base-build/img/ & js/ are for organizing your images and JavaScript code.

!project-files/ is where all files go that will never be uploaded to the server but are related to the project in some aspect. Any general assets, code modules or (what was originally intended for) original stock photography.

The rest of the files mimic the structure of the !base-build/ folder with the exception of mobile/. the mobile/ folder is used for WordPress's plugin WPTouchPro. WPTouchPro is the current mobile plugin used for our mobile phone offering. Other files in the base client-folder/ includes a function.php file for adding specific navigations, widget areas, etc.; screenshot.png a screenshot of the home page themplate; and a few other .php files that are the minimum requirements for the WordPress children themes.

Modular Zones Get in the zone, Autozone.

Zen Garden

The idea behind our Modular websites is loosely based of a brilliant piece of work by Dave Shea called the CSS Zen Garden. The idea was with one set of HTML one could create just about any visual web site they wanted to. Go to the CSS Zen Garden for more details and examples.

Modular

The goal with our Modular's is to have one snippet set of HTML that can be used as a head start for each of 6 zones. That snippet of HTML is properly labeled, and can be copied and pasted into the document one is working on to get styling with CSS faster!

Zones

The concept of a 'zone' is a horizontal slice, or region of a web site. This horizontal slice is represented by two factors, the six zones are listed below and each zone has multiple versions described with a number. In other words a zone of Header 4 is the header zone, and the 4th option of that zone. One would look in the Header Zone snippets for option 4. The designers based their design off this layout and this HTML snippet will give one a head start on getting the CSS going.

A full list of zones:

  1. Header Zone
  2. Marketing Zone
  3. Buckets Zone
  4. Content Zone
  5. Footer Zone
  6. Quick Contact Zone
Note: Not all Zones are used on every project. To be truly flexible some zones may be removed, see the specific project for details.

Naming Conventions Order in the court!

What? Where? Why?

We believe in using thoughtful, mindful, meaningful markup. We are creative beings and appreciate the work we do so why not build some standards to make things easy for others to figure out when they are reading our code?

Also note that the web does not like spaces, so instead replace spaces with dashes. Use dashes between each word in your naming conventions and lowercase lettering.

Where are we using this?

Everywhere. Let's list it out to make it even more clear:

  • HTML
  • CSS
  • JavaScript
  • Photoshop Slices
  • Media
  • Folders

Three Questions

Remember these three questions whenever you are naming something and you'll start to see things fall into place.

  1. What is it?
  2. Where is it?
  3. How would you describe it?
Note: One doesn't have to answer all three, sometimes the first two are specific enough. Use your best judgement.
Sometimes the first two questions are enough. screenshot-naming-convention.png is enough for this image to be descriptive.
